Ticketing System (osTicket)
osTicket · LAMP Stack · Ubuntu · VirtualBox
Completed

Self-hosted osTicket on a Linux VM running a full LAMP stack inside VirtualBox. Configured SLAs, escalation rules, departments, and email piping to simulate a real enterprise helpdesk workflow end-to-end — mirroring how IT support teams manage incoming incidents, service requests, and change tickets in production environments.

IT Support osTicket LAMP Stack Ubuntu Apache MySQL PHP VirtualBox SLA Management
Objectives
  • Deploy and configure a self-hosted ticketing system to simulate a real IT helpdesk environment
  • Understand the full lifecycle of an IT ticket — from submission and triage to resolution and closure
  • Practice configuring SLAs, priority levels, and escalation rules to mirror enterprise support workflows
  • Set up department routing so tickets are automatically assigned to the correct team (IT, Network, Security)
  • Configure email piping so incoming emails automatically generate tickets in the system
  • Learn Linux server administration skills: Apache, MySQL, PHP, and file permission management on Ubuntu
What Was Built
  • Deployed Ubuntu Server 22.04 LTS VM in VirtualBox with a Bridged Network adapter for LAN access
  • Installed and configured a full LAMP stack — Apache2, MySQL 8, PHP 8.1 — as the osTicket backend
  • Downloaded, installed, and configured osTicket v1.18 including database creation and web installer setup
  • Created 3 departments: IT Support, Networking, and Cybersecurity — each with dedicated agents and queues
  • Configured 4 SLA tiers: Critical (1hr), High (4hr), Normal (8hr), Low (24hr) with auto-escalation on breach
  • Built help topics for Password Reset, Hardware Issue, Software Request, Network Outage, and Security Incident
  • Set up email piping via fetchmail so emails sent to helpdesk@blaborte.local auto-create tickets
  • Created agent accounts with role-based access: Admin, Manager, and Technician permission levels
  • Tested the full ticket lifecycle: open → assigned → in-progress → resolved → closed, with internal notes and canned responses

Step-by-Step Setup Guide
01
Create the Ubuntu VM in VirtualBox
In VirtualBox, create a new VM — Type: Linux, Version: Ubuntu (64-bit). Assign at least 2 vCPUs and 2GB RAM (4GB recommended). For networking, use Bridged Adapter so the VM gets an IP on your local network and you can access the osTicket web portal from your host machine's browser.
Download Ubuntu Server 22.04 LTS ISO from ubuntu.com. Attach it under VM Settings → Storage → add optical drive. During install, select "Install OpenSSH server" for easy remote access.
02
Update the System & Install LAMP Stack
After booting into Ubuntu Server, update all packages then install Apache, MySQL, and PHP with the extensions osTicket requires.
sudo apt update && sudo apt upgrade -y sudo apt install apache2 mysql-server php php-mysqli php-gd php-imap php-xml php-mbstring php-intl php-apcu libapache2-mod-php -y sudo a2enmod rewrite sudo systemctl restart apache2
03
Secure MySQL & Create the Database
Run the MySQL secure installation wizard, then log in and create a dedicated database and user for osTicket.
sudo mysql_secure_installation sudo mysql -u root -p # Inside MySQL shell: CREATE DATABASE osticket; CREATE USER 'osticket'@'localhost' IDENTIFIED BY 'YourPassword!'; GRANT ALL PRIVILEGES ON osticket.* TO 'osticket'@'localhost'; FLUSH PRIVILEGES; EXIT;
04
Download & Install osTicket
Download the latest osTicket release, extract it to Apache's web root, and set the correct file permissions.
cd /tmp wget https://github.com/osTicket/osTicket/releases/download/v1.18.1/osTicket-v1.18.1.zip sudo apt install unzip -y unzip osTicket-v1.18.1.zip -d osticket sudo cp -r osticket/upload /var/www/html/osticket sudo cp /var/www/html/osticket/include/ost-sampleconfig.php \ /var/www/html/osticket/include/ost-config.php sudo chmod 0666 /var/www/html/osticket/include/ost-config.php sudo chown -R www-data:www-data /var/www/html/osticket
05
Configure Apache Virtual Host
Create an Apache config file for osTicket so it serves correctly from the web root.
sudo nano /etc/apache2/sites-available/osticket.conf
Paste this into the file: ServerName your-server-ip, DocumentRoot /var/www/html/osticket, then run: sudo a2ensite osticket.conf && sudo systemctl reload apache2
06
Run the Web Installer
Open your browser and navigate to your server's IP address. The osTicket setup wizard will guide you through connecting to the database and creating the admin account.
http://<your-server-ip>/osticket/setup/
After setup completes, delete the /setup directory and lock down ost-config.php: sudo rm -rf /var/www/html/osticket/setup && sudo chmod 0644 /var/www/html/osticket/include/ost-config.php
07
Create Departments & Help Topics
Log into the Admin Panel (yourip/osticket/scp). Go to Admin Panel → Agents → Departments and create IT Support, Networking, and Cybersecurity. Then go to Manage → Help Topics and add topics like Password Reset, Hardware Issue, and Network Outage.
Help Topics determine which department a ticket routes to automatically. Set a default SLA and priority per topic for consistent triage.
08
Configure SLA Plans
Go to Admin Panel → Manage → SLA Plans. Create your tiers with grace periods and schedules. SLAs define how quickly a ticket must be responded to and resolved before it auto-escalates.
SEV-A (Critical): Grace Period 1hr — 24/7 SEV-B (High): Grace Period 4hr — 24/7 SEV-C (Normal): Grace Period 8hr — Business Hours SEV-D (Low): Grace Period 24hr — Business Hours
09
Create Agent Accounts & Assign Roles
Go to Admin Panel → Agents → Add New Agent. Create agents for each department and assign roles (Admin, Manager, Technician). Roles control what agents can see, edit, assign, and close.
Create at least one end-user account under the Client Portal to simulate submitting and tracking tickets from the user's perspective.
10
Test the Full Ticket Lifecycle
Submit a test ticket via the client portal or by sending an email (if piping is configured). Then log in as an agent and walk through the full workflow: acknowledge → assign → update → add internal notes → resolve → close. Verify the SLA timer, escalation, and department routing all work correctly.
Try deliberately breaching an SLA to confirm the escalation alert fires. Check Admin Panel → Dashboard for ticket stats and agent performance metrics.
